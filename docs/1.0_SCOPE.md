# StatsPackage 1.0 Scope

This document defines the current objective for moving `StatsPackage` from post-`0.1.0` stabilization to a true `1.0.0` release.

## Current State

- Current public release: `0.1.4`
- The package installs successfully and has been used effectively by classmates.
- Core function families are already present: confidence intervals, hypothesis tests, width planning, power, required sample size, descriptive shape, and chi-square / categorical-data methods.
- The package is already release-quality, but the public API is still treated as stabilizing rather than permanently frozen.

## 1.0 Objective

The goal of `1.0.0` is to declare that the public API is stable for the package's intended use: reusable introductory statistical inference workflows.

That means:

- exported functions have stable names and argument meanings
- defaults are intentional and unlikely to change casually
- printed summaries and returned result objects are consistent across families
- the README, vignette, pkgdown site, cheat sheet, and tests all describe the same behavior

## In Scope For 1.0.0

- freeze the current public API unless a clear correctness issue requires a change
- fix any remaining documentation inconsistencies with actual function behavior
- strengthen regression coverage around public behavior and representative examples
- preserve the package's core design conventions:
  - readable printed output when `quiet = FALSE`
  - silent use when `quiet = TRUE`
  - invisible, classed return objects with computational details
- keep the package focused on general, reusable inference workflows rather than assignment-specific wrappers

## Out Of Scope Before 1.0.0

- broad feature expansion without a strong reusable-statistics justification
- assignment-specific helper wrappers
- breaking renames or casual default changes
- replacing stable functions wholesale when a targeted patch is enough

## 1.0 Entry Criteria

`StatsPackage` is ready for `1.0.0` when all of the following are true:

- the package has been exercised through real use and the API feels frozen
- `devtools::test()` passes
- `devtools::check()` passes cleanly
- CI, lint, coverage, and pkgdown checks are green
- cheat-sheet and smoke-test workflows pass
- install from GitHub works in a clean session
- no known mathematical correctness issues remain in exported functions
- the README, vignette, cheat sheet, and site are aligned with actual package behavior

## Release Approach

- continue using `0.1.x` releases for stabilization patches
- use those patch releases for bug fixes, doc clarification, QA tightening, and minor pedagogical cleanup
- move to `1.0.0` only when changes are mostly hardening work rather than API design work
- after `1.0.0`, treat argument meanings, defaults, and return-object conventions as stable commitments

## Candidate Future Tasks

- consider adding optional confidence-interval output to `z_test_mu()` and `t_test_mu()`
- if implemented, use `conf.level = 1 - alpha` and map one-sided alternatives to one-sided bounds
- reuse existing CI-family logic where possible so test output and CI output cannot drift apart
- consider adding a raw-data summary helper that converts numeric vectors into reusable summary statistics
- prefer that helper over forcing every summary-based inference function to accept raw-vector inputs directly
- evaluate adding a raw-data `wilcox_test()` function as a standalone nonparametric hypothesis test
- if added, keep the first version limited to the test itself rather than immediately adding matching power, required-`n`, or CI functions
- evaluate adding a small bootstrap family for raw-data inference, designed to be more pedagogical and focused than `boot`
- if added, start with a narrow CI-oriented scope for common statistics rather than a general-purpose bootstrap framework
- consider adding `fisher_test_2x2()` as a small-sample companion to `chisq_table()`
- consider adding `pred_int_mu()` for prediction intervals on a future observation
- consider adding correlation inference helpers such as `cor_test()` and `ci_cor()` if correlation is part of the stable course scope
- consider adding basic regression-inference helpers only if regression is clearly in scope for the package
- consider adding effect-size helpers such as `cohen_d()` or simple `2 x 2` effect measures if they support the package's teaching goals

## Decision Rule

For proposed work before `1.0.0`, ask:

Does this make the current toolkit more reliable, consistent, and easier to trust?

- If yes, it likely belongs in the `1.0.0` hardening scope.
- If it mainly expands scope, it should usually wait until after `1.0.0`.
